# AgentWeb Unified Research RAG Constraints

This document defines the mandatory architecture and engineering boundaries for AgentWeb.
When code, a design proposal, or a legacy document conflicts with this document, update this
document first and then change the implementation.

## 1. Product Scope

AgentWeb is one single-crystal-growth research workbench. It provides three user-facing
capabilities through one application:

1. RAG dialogue and research assistance.
2. Literature and experimental-record retrieval.
3. Formula-conditioned growth-route prediction.

The user-facing product is not split into independent search, inference, and RAG products.
Retrieval and prediction are internal capabilities. A Chat answer follows a retrieval-first
evidence-selection policy: it uses sufficient real records, or, only after a completed but
insufficient retrieval, it may use a model prediction. It must not combine the two kinds of
evidence in one answer.

## 2. Active Application Boundary

The active application boundary is:

```text
apps/rag-platform/       the only user-facing web application
services/rag-api/        the only public business API and LangGraph entrypoint
```

The following directories are legacy placeholders. No new code, route, package dependency,
or deployment entry may be added to them:

```text
apps/search-platform/
apps/inference-platform/
apps/portal/
services/search-api/
services/inference-api/
```

They must be removed in a dedicated workspace-cleanup change after all package references
and documentation references have been migrated. Their existence does not establish a
second product or service boundary.

## 3. Internal Service Structure

`services/rag-api` must keep clear internal capability boundaries:

```text
api/          HTTP and SSE endpoints only
agent/        LangGraph state, routing, and workflow nodes
retrieval/    Milvus hybrid retrieval and retrieval planning adapters
prediction/   model registry, input validation, inference, and output normalization
evidence/     evidence contracts, provenance, sufficiency assessment, and selection rules
memory/       short- and long-term memory persistence
accounts/     authenticated identity and authorization dependencies
db/           PostgreSQL persistence access
ingestion/    growth-record normalization and import
```

The dependency rules are mandatory:

1. Frontend code must call only `rag-api`; it must not connect to PostgreSQL, Milvus,
   the prediction model, or an LLM directly.
2. LangGraph nodes must call `RetrievalService`, `PredictionService`, `MemoryStore`, and
   other explicit interfaces. Nodes must not directly import `pymilvus`, `torch`, or a
   database driver.
3. `retrieval/` owns all Milvus access.
4. `prediction/` owns model loading, formula normalization, constrained decoding,
   post-processing, and model-version reporting.
5. `evidence/` owns the distinction, retrieval-sufficiency assessment, and mutually exclusive
   selection rules between literature evidence and model-prediction evidence. It must not
   merge their scores or construct a combined evidence pack.
6. Stable reusable types may be placed in a shared package. RAG-specific business logic
   must remain inside `services/rag-api`.

## 4. Research Evidence Rules

The system has at least two evidence kinds:

```text
literature_record
  A paper-derived or historical experimental record retrieved from Milvus.

model_prediction
  A route candidate generated by a versioned local prediction model.
```

These evidence kinds are never interchangeable. Except for direct, clarification, and error
responses, every Chat answer selects exactly one of them as its `selected_evidence_kind`.

Literature evidence must include a `record_id`, source text, and DOI when available.
Prediction evidence must include `prediction_run_id`, `model_id`, `model_version`, input,
structured output, and validation warnings.

The system must never:

1. Present a prediction as a literature fact.
2. Invent a DOI, source text, or record ID for a model prediction.
3. Treat a beam-search ranking score as an experimental success probability or calibrated
   confidence.
4. Store prediction output in Milvus as if it were a source record.
5. Write every prediction automatically into long-term memory.
6. Run the predictor after retrieval has been judged sufficient.
7. Pass weak, mismatched, or incomplete retrieved records into a prediction answer as if they
   supported the model route.
8. Treat a Milvus timeout, connection error, or other retrieval-service failure as evidence
   that the scientific corpus contains no relevant record.

For a literature answer, the visible evidence section contains only real records and their
citations. For a `prediction_fallback` answer, it contains only the versioned model output,
warnings, and the retrieval-fallback reason. For an `explicit_prediction` answer, it contains
the same model provenance and warnings without a fallback claim. Neither kind of prediction
answer has literature citations unless the user starts a separate retrieval request.

### 4.1 Retrieval Sufficiency and Fallback

Retrieval completion and retrieval sufficiency are different concepts. Before selecting an
answer source, the system must produce a structured `RetrievalOutcome`:

```text
status: sufficient | empty | insufficient | invalid_request | unavailable
reason_codes: list[str]
requested_slots: list[str]
covered_slots: list[str]
usable_record_ids: list[str]
fallback_allowed: bool
```

`sufficient` requires usable, traceable records that match the material and any explicit
growth-method constraint, and that cover the fields needed to answer the question. A returned
Milvus hit, a high vector score, or a non-empty result list alone is not sufficient.

`empty` and `insufficient` may allow model fallback only when the user's task is to obtain a
candidate growth route and an unambiguous, parseable formula is available from the request or
the active conversation context. Candidate-route requests include direct operational wording
such as "Mn3GaN怎么做", "我要长 Mn3GaN 单晶", "如何制备 Mn3GaN", and a follow-up such
as "那你推测一个" after one active formula is established. No further confirmation is required:
after a completed insufficient retrieval, LangGraph autonomously invokes the local model. The
model supports only formula-conditioned Flux/CVT route generation; it cannot replace missing
factual answers about literature temperatures,
atmospheres, or other record-specific conditions.

`unavailable` never allows automatic fallback. It must report a retrieval-service failure.
`invalid_request` and a missing or invalid formula must lead to clarification or a bounded
no-evidence response, not an invented model input.

The initial sufficiency gate must be deterministic and inspect record identity, normalized
formula/material match, normalized growth-method match, requested field coverage, and source
integrity. An LLM may explain a completed decision but must not override these minimum gates.

### 4.2 Explicit Prediction

The Predict workspace and `POST /api/rag/predict` are explicit model invocations. They may
run `PredictionService` directly after input validation, without first searching Milvus. This
does not make the resulting output retrieval evidence and does not change the Chat
retrieval-first policy.

## 5. Prediction Model Rules

The initial deployed predictor is the versioned growth-route Transformer bundle at:

```text
services/rag-api/models/growth-route-transformer/v2.0.0/
```

The bundle contains only runtime assets: model checkpoint, inference source snapshot,
vocabularies, decoder metadata, reactant-element mapping, and normalization mappings.
Training data, raw data, and experimental reports must not be copied into the RAG runtime
bundle.

Every deployed model bundle must include:

```text
MANIFEST.json
MODEL_CARD.md
checkpoint and runtime assets
```

`MANIFEST.json` must record the model ID, version, artifact digest, task contract, supported
methods, dependency requirements, and evaluation summary.

The v2.0.0 model accepts only a formula and returns up to three constrained route candidates.
It supports only `Flux` and `CVT`. It is not conditionally trained on user constraints such
as a maximum furnace temperature. The system may filter or warn about generated routes using
such constraints, but must not claim that the model predicted under them.

The predictor must reject or warn on:

1. Empty or unparsable formulas.
2. Unknown formula tokens.
3. Unsupported growth methods.
4. Incomplete structured input required by a future model version.
5. Route candidates that fail output validation.

## 6. LangGraph Capability Routing

The graph must route a question by entry mode and selected evidence source. It must not have a
parallel `retrieval_and_prediction` path.

```text
direct_answer          a non-evidence response
retrieval_first        Chat request that requires research evidence
literature_answer      sufficient retrieval selected as the sole evidence source
prediction_fallback    completed but empty/insufficient retrieval selected model fallback
explicit_prediction    Predict workspace or direct API model invocation
clarify                missing task detail or prediction input
retrieval_unavailable  retrieval infrastructure failed; no automatic model fallback
```

The expanded Chat workflow must preserve the existing bounded memory behavior and use this
ordered decision path:

```text
prepare_turn
-> load_context
-> load_long_memory
-> understand_query
-> route_entry
-> plan_retrieval
-> retrieve_records
-> assess_retrieval_sufficiency
-> sufficient          -> build_literature_evidence -> answer_from_literature
-> empty/insufficient  -> resolve_formula -> validate_prediction_input
                       -> run_prediction -> validate_prediction_result
                       -> answer_from_prediction
-> unavailable         -> report_retrieval_unavailable
-> update_memory
-> finalize_response
-> compact_persistent_state
```

`answer_from_literature` receives only `LiteratureEvidence` and may emit citations.
`answer_from_prediction` receives only `PredictionEvidence`, must expose its fallback reason,
and must emit no literature citations. Retrieval diagnostics may be retained for audit but
must not be supplied to the prediction-answer prompt as supporting evidence.

Prediction input validation must occur before model execution. If a request lacks the formula,
asks for a method outside the model contract, or asks for a factual answer that the model
cannot produce, the graph must ask a clarification question or return a bounded explanation
rather than inventing model input.

## 7. Storage Boundaries

PostgreSQL is the system of record for:

```text
users and authenticated sessions
conversation metadata
LangGraph checkpoint-session mappings
long-term memory items and memory jobs
prediction model registry
prediction runs and their structured outputs
```

Milvus stores only public scientific corpus records and their sparse/dense retrieval index.
It does not store user memory, login data, authentication sessions, or model predictions.

Prediction runs must be scoped by trusted `user_id`. The product intentionally does not add
`project_id` or `laboratory_id` to the memory or prediction-run ownership model.

## 8. Authentication and User Isolation

Production requests must derive `user_id` from an authenticated server-side session. The
browser must not supply a trusted `user_id` in a RAG request.

The web product uses an email-and-password account model. Email is a normalized, unique account
identifier only; this product intentionally does not send, require, or validate email ownership
codes. A successful first login with an unregistered email and a valid password creates the
account. A later login with the same email must verify against its stored Argon2id password hash.

Authentication sessions must use opaque, server-stored token hashes and HttpOnly cookies. Raw
passwords and raw session tokens must never be persisted. Login attempts must be rate-limited.

Development CLI mode may accept an explicit `--user-id` and `--session-id` only for local
memory-isolation tests. Those values are not authentication and must be rejected by the
production web API.

All of the following are scoped by the authenticated user identity:

```text
long-term memory
short-term checkpoint session mapping
conversation history
prediction runs
saved evidence and feedback
```

## 9. API and UI Rules

The public API base is `/api/rag/`. The planned capability endpoints are:

```text
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me

POST /api/rag/chat/stream
POST /api/rag/retrieve
POST /api/rag/predict
GET  /api/rag/sessions
POST /api/rag/sessions
PATCH /api/rag/sessions/{session_id}
DELETE /api/rag/sessions/{session_id}
GET  /api/rag/sessions/{session_id}/messages
GET  /api/rag/prediction-runs
```

`/api/rag/predict` and the LangGraph prediction node must use the same `PredictionService`.

The single web application must provide three coherent workspaces:

```text
Chat             retrieval-first research assistance with controlled prediction fallback
Evidence         direct retrieval, filters, source records, and DOI inspection only
Predict          explicit formula input, model candidates, warnings, and run history
```

For a single Chat response, the UI must render one selected evidence type, never a combined
literature-and-prediction panel. A literature response shows real records and citations. A
prediction fallback shows the failed-sufficiency reason, model ID/version, supported scope,
warnings, and prediction-run reference. The Evidence workspace never turns an empty search
into a model prediction.

## 10. Deployment and Resource Rules

The public 1 GiB server is limited to HTTPS termination, static frontend delivery, reverse
proxying, and the private tunnel endpoint. It must not run Milvus, PostgreSQL, the local
embedding model, or the route-prediction model.

The ThinkPad Ubuntu host runs:

```text
rag-api
PostgreSQL + pgvector
Milvus stack
MiniLM embedding runtime
growth-route prediction runtime
memory worker
```

The prediction runtime must be versioned and load at most one model instance per configured
worker process. CPU inference must have a bounded concurrency setting. Batch scans or long
parameter sweeps must become tracked background jobs rather than blocking the chat stream.

## 11. Configuration and Dependencies

LLM and embedding providers remain behind their existing provider interfaces. The local
route predictor is accessed only through `PredictionService`; it is not an
OpenAI-compatible LLM provider.

Prediction-specific dependencies must be isolated in a `prediction` optional dependency
group. The initial bundle requires PyTorch, NumPy, and pymatgen. Prediction configuration
must use the `PREDICTION_` prefix and contain no secrets.

## 12. Verification and Documentation

Every prediction-model integration must provide:

1. Checkpoint digest verification.
2. Formula parsing tests.
3. Deterministic inference smoke tests for known formulas.
4. Route schema and element-coverage validation tests.
5. Tests that model evidence cannot masquerade as literature evidence.
6. User-isolation tests for prediction-run records.
7. A model-card update when model assets or evaluation metrics change.
8. A test that sufficient retrieval does not invoke `PredictionService`.
9. A test that an empty or insufficient completed retrieval invokes one validated prediction
   only when the request is prediction-eligible.
10. A test that retrieval-service unavailability never invokes `PredictionService`.

Architecture changes must update:

```text
README.md
docs/project-constraints.md
docs/unified-research-platform-design.md
docs/rag-langgraph-design.md
docs/rag-two-layer-memory.md when memory behavior changes
```
